# /confirm - 设计确认

## 描述

展示生成的设计稿，引导用户确认设计方向，记录确认结果和反馈。

## 前置条件

- **必须存在**：`design-assets/` 目录
- **输入文件**：`design-assets/` 下的设计稿文件

## 输出

- **输出文件**：`Design-Confirmation.md`
- **包含内容**：
  - 确认时间
  - 确认状态（已确认/部分确认/待修改）
  - 已确认文件列表
  - 待改进项
  - 反馈详情

## 工作流程

```
用户输入 /confirm
  ↓
主控检测 design-assets/ 目录是否存在
  ↓
展示所有设计稿文件
  ↓
引导用户确认设计方向
  ↓
收集反馈和修改意见
  ↓
生成 Design-Confirmation.md
```

## 确认选项

| 选项 | 描述 | 后续操作 |
|------|------|----------|
| 全部确认 | 所有设计稿都符合预期 | 进入 /dev 开发 |
| 部分确认 | 部分设计需要修改 | 标记未确认项，可继续开发 |
| 需要修改 | 设计稿需要修改 | 使用 /art 重新生成 |
| 重新生成 | 对设计不满意，重新生成 | 使用 /art 重新生成 |

## 确认状态

| 状态 | 描述 | 后续操作 |
|------|------|----------|
| 已确认 | 所有设计稿均已确认 | 进入 /dev 阶段 |
| 部分确认 | 部分设计稿已确认 | 可进入开发，标记未确认项 |
| 待修改 | 设计稿需要修改 | 使用 /art 重新生成 |

## 相关指令

| 指令 | 描述 | 关系 |
|------|------|------|
| `/art` | 生成设计稿 | 前置步骤 |
| `/dev` | 开始代码开发 | 下一步 |

## 后续操作

设计确认后：
1. **已确认**：使用 `/dev` 开始代码开发
2. **待修改**：使用 `/art` 重新生成
3. **部分确认**：可进入开发，但需注意未确认项
